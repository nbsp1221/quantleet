## Why This Document Exists

`quantleet` is developed in an agent-first workflow. AI coding agents can
produce, update, and reshape code much faster than a human-only workflow.

That changes the highest-return quality question.

For a repository that already has strong linting, strict typing, coverage,
duplicate-code detection, dead-code detection, dependency declaration checks,
and architecture docs, the most important next question is not only whether
basic implementation mistakes are caught. It is whether the repository can
detect:

- weak tests that look adequate because they pass and improve coverage
- architecture drift that emerges because agents take the shortest local path
- repeated shortcut patterns that should become fitness functions

This document captures the current research-backed position. It is not itself a
hard gate. Use the admission rule in
[architecture-governance.md](architecture-governance.md) before promoting any
item here into `repo_check`, structure tests, or CI.

## Hypothesis Under Review

The working hypothesis is:

1. Frontier coding models are already strong enough that many basic coding
   mistakes are less likely to be the highest-value failure mode once standard
   gates are installed.
2. Basic failures such as formatting, syntax, imports, typing, and simple test
   breakage are comparatively easy for humans and machines to detect.
3. The higher-risk agent failure modes are test-quality weakness and
   architecture-quality drift.

Current conclusion:

- The hypothesis is mostly supported for this repository.
- Security and supply-chain gates still matter, but they are not the most direct
  answer to the agent-specific risks already visible in this codebase.
- The next highest-return investment is likely to be test-quality gates,
  mutation baselines, and architecture fitness functions.

## Evidence From 2025-2026 Research

### Agentic CI Reliability

An MSR 2026 study of 11,771 GitHub pull requests compared 7,619 agentic PRs with
4,152 human-authored PRs. The study found that AI-authored CI fixes were much
faster in median time-to-fix, but agents also introduced a large share of CI
failures and were less involved in repairing them.

Implication for `quantleet`:

- CI and local gate parity still matter.
- However, once basic CI catches syntax, test, type, and lint failures, the
  remaining agent-specific risk shifts toward whether the tests and
  architecture rules are meaningful.

Source:

- <https://2026.msrconf.org/details/msr-2026-mining-challenge/25/On-the-Reliability-of-Agentic-AI-in-Continuous-Integration-Pipelines>

### LLM-Generated Test Weakness Under Evolution

A 2026 study on LLM-based test generation under software evolution found that
LLM-generated suites can achieve strong baseline coverage on the original
program, but degrade when programs evolve. Under semantic-altering changes, the
reported pass rate and branch coverage fell materially, and the authors
concluded that generated tests can rely heavily on surface-level cues rather
than durable semantic understanding.

Implication for `quantleet`:

- Line and branch coverage are necessary but insufficient.
- Diff coverage is valuable, but it still needs mutation and semantic scenario
  pressure.
- Tests generated by agents should be evaluated by whether they kill meaningful
  behavior changes, not only by whether they execute code.

Source:

- <https://arxiv.org/abs/2603.23443>

### AI-Generated Technical Debt In The Wild

A 2026 large-scale empirical study analyzed 302.6k verified AI-authored commits
from 6,299 GitHub repositories. The study found that code smells were the most
common introduced issue category, and that a non-trivial share of tracked
AI-introduced issues survived into the latest repository revisions.

Implication for `quantleet`:

- The risk is not only broken code.
- The higher-volume risk is maintainability drift that remains after tests pass.
- Architecture fitness functions should focus on repeated drift patterns, not
  generic style preferences.

Source:

- <https://arxiv.org/abs/2603.28592>

### Coding-Agent Security Debt

A 2025 systematic analysis of LLM coding agents found insecure actions in a
meaningful share of evaluated agent trajectories, with large variation between
models and mitigation strategies.

Implication for `quantleet`:

- Security gates should not be dismissed.
- They are still a lower-cost safety layer, but for this repository they are a
  second-order priority behind test-quality and architecture-fitness gates
  because the existing baseline already catches many basic implementation
  failures.

Source:

- <https://arxiv.org/abs/2507.09329>

### Vibe-Coding And Evaluation Practice

FSE 2026 work on vibe coding describes a spectrum from users who rely almost
entirely on AI to users who inspect and adapt generated output. Across that
spectrum, evaluation practice and trust calibration shape outcomes.

Implication for `quantleet`:

- The repository should not assume that every agent-generated change received
  deep human inspection.
- Durable docs, executable contracts, and fitness functions are the scalable
  substitute for chat-only memory.

Source:

- <https://spideruci.org/publication/chou-rolling-2026/>

## Current Repository Position

The repository already has stronger-than-average local mechanical gates:

- Ruff formatting and linting, including complexity checks
- strict `mypy` over `src`
- pytest pass/fail lane
- branch coverage with a full-project floor
- changed-line coverage
- coverage baseline ratchet
- jscpd duplicate-code detection
- Vulture dead-code detection
- Deptry dependency declaration checks
- package build and Twine artifact validation
- notebook validation
- repository architecture and docs checks
- performance checks for runtime-sensitive paths
- aggregate mutation score testing for the `trading` and `backtest` kernels

This means the next agent-specific quality problem is not primarily
"install more basic linters." The more important question is whether the
existing gates are measuring the right failure modes.

## Test-Quality Gaps

Current strengths:

- Coverage is branch-aware.
- Changed-line coverage prevents untested new source lines.
- A baseline ratchet prevents slow coverage decay.
- Mutation testing exists for the `trading` and `backtest` kernels.
- Runtime-sensitive work has a stronger check lane.

Remaining gaps:

1. Mutation testing is too narrow for the full agent risk surface.
   - Current mutation scope is `src/quantleet/trading`.
   - High-risk behavior also lives in `backtest`, `research`, data adapters,
     order activation, sizing, and execution-model code.
2. Mutation results are not yet a ratcheted policy.
   - A run can reveal survivors, but the repository does not yet define a
     committed survivor baseline or fail-on-regression rule.
3. Test smell detection is mostly review-led.
   - Weak assertions, broad exception assertions, integration-test mocking,
     overpowered helpers, and snapshot-only checks can pass coverage gates.
4. Property and metamorphic tests are underused.
   - The domain has natural invariants that are stronger than many example
     tests.

High-value test-quality candidates:

- mutation baselines per bounded risk area
- survivor ratchet similar to the coverage baseline
- integration-test mocking restrictions with explicit allowlists
- AST checks for weak assertions and broad `pytest.raises(Exception)` use
- property tests for cash, fill, order-sizing, slippage, fee, and source-versus-bars invariants
- metamorphic tests for monotonic fee/slippage effects and equivalent input paths

## Architecture-Quality Gaps

Current strengths:

- `quantleet` has explicit bounded contexts.
- Cross-domain import rules already exist in repository tooling.
- Structure tests protect public docs, package metadata, test layout, and
  several current contract surfaces.
- The workflow already says repeated drift should be promoted into checks.

Remaining gaps:

1. Import rules do not catch all shortcut architecture.
   - Agents can introduce helpers in plausible locations that preserve imports
     but blur responsibility.
2. Test support code can hide product architecture weakness.
   - A support helper can make tests concise while centralizing behavior that
     should belong to the product code.
3. Public API drift is only partially protected.
   - Explicit facade snapshots can make accidental surface expansion or
     contraction easier to detect.
4. Complexity and fan-out are only partially guarded.
   - Ruff C90 catches function complexity, but module-level responsibility
     growth and dependency fan-out need separate pressure.

High-value architecture-fitness candidates:

- explicit architecture fitness tests for context dependency direction
- module fan-out thresholds for high-risk runtime modules
- public API snapshot checks for exported package surfaces
- checks limiting where test support modules may import from and be imported by
- circular dependency detection at package-context granularity
- protected-module rules for `trading` and order-runtime semantics
- review-triggered promotion for repeated shortcut patterns

Candidate tools to evaluate:

- `pytest-archon`, which expresses architecture dependency rules as pytest tests
  and supports forbidden dependency checks.
- `import-linter`, which supports contracts such as layers, independence,
  protected modules, and acyclic siblings.
- Custom extensions to `quantleet._repo_tools`, which may fit best when the
  check is repo-specific and should remain legible to agents.

Sources:

- <https://pypi.org/project/pytest-archon/>
- <https://github.com/seddonym/import-linter>

## Supply-Chain And Security Gates

Security and supply-chain gates remain useful, especially because agents can add
dependencies or edit workflow files quickly.

Reasonable future candidates:

- `pip-audit` for Python dependency vulnerabilities
- GitHub CodeQL for code scanning
- GitHub dependency review for dependency changes
- `zizmor` or `actionlint` for GitHub Actions issues

However, for the current repository maturity, these are not the highest-return
answer to the agent-specific quality concern under review. They are broad safety
rails. Test-quality and architecture-fitness gates target the more likely
long-term agent failure mode: plausible, passing, maintainability-eroding code.

## Priority Recommendation

Recommended order for future gate work:

1. Test-quality gate improvements
2. Architecture fitness functions
3. Mutation baseline and survivor ratchet expansion
4. CI parity with the local hard gate
5. Supply-chain and workflow-security gates

This order is specific to the current repository state. If the repository starts
adding many dependencies, deployment automation, credentials, or live execution
scope, security and supply-chain gates should move up.

## Check Admission Notes

Before promoting any recommendation here into a hard gate, document:

1. protected behavior
2. measured proxy
3. likely gaming vector
4. decision the check will change
5. revalidation or removal condition

The strongest first candidates are narrow and objective:

- reject broad `pytest.raises(Exception)` in tests
- reject integration-test mocking outside allowlisted support seams
- enforce public API export snapshots
- ratchet mutation survivors for one bounded runtime area

Avoid broad subjective metrics such as "architecture quality score" unless they
are decomposed into explicit, review-proven failure modes.
