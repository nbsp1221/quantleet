# Quantcraft 에이전트 개발 운영 가이드

> 상태: advisory 운영 메모
> 현재 권위 문서:
> - `AGENTS.md`
> - `ARCHITECTURE.md`
> - `docs/design-docs/index.md`
> - `docs/product-specs/index.md`
>
> 이 문서는 에이전트 개발 운영에 관한 개인용 해설 메모다.
> 현재 저장소 규칙이나 아키텍처 truth를 직접 정의하지 않는다.

## 목적

이 문서는 `quantcraft`를 앞으로 에이전트 중심으로 개발할 때, 어떤 원칙으로 기능 개발과 하네스 강화를 함께 진행할지 정리한 개인용 운영 문서다.

핵심 목표는 다음 두 가지다.

- 내가 매번 코드를 직접 다 읽고 관리하지 않아도 되게 만들 것
- 금융/퀀트 코드 특성상 AI가 저장소를 오염시키거나 위험한 방향으로 구조를 망치지 못하게 할 것

이 문서는 OpenAI의 "Harness engineering: leveraging Codex in an agent-first world" 글에서 얻은 원칙을 `quantcraft` 상황에 맞게 해석한 운영 메모다.

원문:

- https://openai.com/index/harness-engineering/

## 가장 중요한 결론

처음부터 모든 규칙과 하네스를 완성형으로 다 정할 수는 없다.

대신 이렇게 해야 한다.

1. 작동 가능한 최소 하네스를 먼저 만든다
2. 실제 기능 개발을 하면서 반복적으로 드러나는 문제를 하네스로 승격시킨다
3. 금융 핵심 도메인은 더 느리게, 더 강한 검증 아래에서 확장한다

즉, `기능 개발`과 `하네스 강화`는 따로 가는 게 아니라 같이 가야 한다.

## OpenAI 문서에서 배운 운영 방식

OpenAI도 처음부터 완벽한 구조를 다 정해놓고 시작한 것은 아니다.

대신 다음 방식으로 저장소를 발전시켰다.

- 빈 저장소에서 시작했다
- 초반에는 진행이 느렸다
- 문제를 모델 성능보다 환경과 구조의 부족으로 봤다
- 문제를 발견할 때마다 그것을 저장소 규칙으로 흡수했다

이때 저장소 규칙으로 흡수하는 방식은 대체로 다음과 같다.

- 문서 업데이트
- 구조 테스트
- 커스텀 린트
- 품질 점수 추적
- 정기 cleanup 작업

즉, 버그나 리뷰 코멘트가 반복되면 "사람이 기억해서 조심"하는 게 아니라, 저장소가 그 실수를 못 하게 바꾸는 식이다.

## Quantcraft 현재 상태에 대한 판단

현재 `quantcraft`는 기본 하네스 1단계는 갖춘 상태다.

이미 들어간 것:

- 짧은 `AGENTS.md`
- 구조화된 `docs/`
- `repo-check`
- `Poe` 기반 고수준 개발 태스크
- 테스트 taxonomy
- `unit / integration / structure / smoke` 분리
- live smoke를 기본 test lane에서 분리
- 기본 검증:
  - `uv run poe verify`
  - `pytest`
  - `ruff`
  - `mypy`
  - `build`

이건 본격적인 개발을 시작할 수 있는 최소 baseline으로는 충분하다.

하지만 아직 금융 핵심 도메인을 안심하고 AI에게 맡길 수준은 아니다.

아직 부족한 것:

- `execution / risk / portfolio` 실제 코드 경계
- Tier A 도메인 전용 invariant 테스트
- 더 강한 architecture enforcement
- 품질 점수의 실운영 루프
- CI 또는 원격 검증 체계

## 앞으로의 기본 전략

### 1. 새 도메인에 들어가기 전 최소 가드레일 먼저

예를 들어 `backtest`, `paper`, `execution` 같은 새 영역에 들어갈 때는 기능 구현 전에 최소한 아래는 먼저 깔고 들어간다.

- spec 문서
- 디렉터리 구조
- 테스트 위치 규칙
- 기본 구조 테스트

모든 걸 다 먼저 만들 필요는 없지만, 최소 진입 규칙 없이 바로 코드부터 만들지는 않는다.

### 2. 반복되는 문제는 즉시 하네스로 승격

한 번 나온 문제는 수정으로 끝날 수 있다.

하지만 두 번 이상 반복되면 저장소 규칙으로 바꿔야 한다.

예:

- 테스트가 자꾸 flat하게 생긴다
  - 테스트 taxonomy enforcement 추가
- 패키징/runtime dependency 실수가 반복된다
  - packaging check 강화
- Tier A import 침범이 반복된다
  - architecture checker 강화
- 문서 누락이 반복된다
  - repo-check에 필수 문서 검사 추가

### 3. 도메인별 자율성을 다르게 준다

`quantcraft`의 장기 범위는 넓지만, 모든 영역에 같은 수준의 자율성을 주면 안 된다.

상대적으로 빠르게 진행 가능한 영역:

- `market_data`
- `research`
- `backtest infra`
- 문서/도구/구조 계층

더 보수적으로 진행해야 하는 영역:

- `execution`
- `risk`
- `portfolio`

이 세 영역은 Tier A로 보고, 구조 격리, 테스트, invariant, human gate가 충분히 생기기 전에는 느리게 다룬다.

### 4. 기능 단위마다 산출물 3개를 같이 남긴다

기능 하나를 넣을 때 최소한 같이 남겨야 하는 것은 다음이다.

- spec 또는 plan
- code
- tests / checks

코드만 생기고 나머지가 따라오지 않으면, 그 기능은 저장소 관점에서 미완성이다.

### 5. AI에게 버그 수정만 시키지 말고 저장소도 같이 고치게 한다

앞으로는 "이 버그를 고쳐"만으로 끝내지 않는다.

항상 같이 물어야 한다.

- 왜 이런 문제가 가능했는가?
- 무엇이 없어서 이런 실수가 발생했는가?
- 문서/테스트/체커/구조 중 무엇을 보강해야 재발이 줄어드는가?

즉, 매번 코드만 바꾸는 게 아니라 저장소의 방어력도 같이 높이는 방향으로 간다.

### 6. 개발자 UX 레이어와 패키지 계약은 분리한다

앞으로 `quantcraft`는 두 가지 명령 계층을 같이 유지한다.

- `scripts/`
  - 저장소 로컬 하네스의 저수준 직접 표면
  - 예: `uv run python scripts/repo_check.py`, `uv run python scripts/notebook_validate.py`, `uv run python scripts/live_smoke.py`
- `Poe task`
  - 로컬 개발자가 자주 쓰는 상위 작업 흐름
  - 예: `uv run poe lint`, `uv run poe format`, `uv run poe verify`

중요한 점은 `Poe`가 편의 계층이라는 이유로 저장소 로컬 하네스 계약을 대체하면 안 된다는 것이다.

즉:

- repo-local harness contract는 `scripts/`
- 개발 생산성/작업 orchestration은 `Poe`

이 분리는 앞으로도 유지한다.

## 개발 요청을 다루는 방식

앞으로 기능 개발 요청은 위험도에 따라 다르게 처리한다.

### 저위험 기능

예:

- 시장 데이터 유틸리티
- 문서/구조 정리
- 로컬 도구

운영 방식:

- 바로 구현 가능
- 끝난 뒤 테스트와 문서, 구조 영향 반영

### 중위험 기능

예:

- 백테스트 엔진 구조
- paper trading 흐름
- command surface 확장

운영 방식:

- 설계 문서 먼저
- 구현
- 구조/테스트 보강

### 고위험 기능

예:

- 주문 생성
- 포지션 계산
- 리스크 제한
- 자금/익스포저에 영향 주는 코드

운영 방식:

- 설계 문서 먼저
- 도메인 경계 정의
- invariant 테스트 먼저
- 구현은 마지막

## 실제 작업 순서의 기본 템플릿

앞으로는 아래 흐름을 기본으로 삼는다.

1. 기능 또는 도메인 목표 정의
2. 설계 문서 작성
3. 필요한 최소 하네스 정의
4. 구현 계획 문서 작성
5. 코드 + 테스트 + 문서 동시 변경
6. fresh verification
7. 리뷰/QA
8. 반복 문제를 하네스로 승격

이 흐름을 벗어나서 "일단 코드부터 빨리 만들고 나중에 정리"하는 방식은 금융 코드에서는 금지에 가깝게 봐야 한다.

## 지금부터의 우선순위

현재 상태 기준으로 가장 자연스러운 다음 단계는 다음 순서다.

1. 실제 `src/quantcraft/` 도메인 구조 만들기
2. `execution / risk / portfolio`를 Tier A로 코드 레벨에서 격리
3. Tier A 전용 invariant 테스트 체계 추가
4. `repo-check`와 architecture check 강화
5. 필요하면 CI 추가

즉, 지금은 하네스가 없는 상태는 아니고, 이제는 `하네스 위에서 개발하면서 하네스를 더 강하게 만드는 단계`다.

## 기억해야 할 운영 원칙

- 처음부터 모든 걸 다 정할 수는 없다
- 그렇다고 하네스를 나중에 몰아서 해서는 안 된다
- 최소 하네스를 먼저 만들고 시작한다
- 반복 실수는 반드시 저장소 규칙으로 승격한다
- 금융 핵심 도메인은 속도보다 구조와 검증을 우선한다
- 에이전트는 코드를 쓰는 도구가 아니라, 저장소 자체를 강화하는 도구로도 써야 한다

## 한 줄 요약

`quantcraft`는 이제 기본 하네스 위에서 본격 개발을 시작할 수 있다. 다만 앞으로는 기능을 만들 때마다, 그 기능이 드러낸 반복 문제를 저장소 규칙으로 계속 승격시키는 방식으로 운영해야 한다.
